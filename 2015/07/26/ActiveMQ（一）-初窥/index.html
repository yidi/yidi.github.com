ActiveMQ（一）-初窥

发表于 2015-07-26   |   分类于 JEE   |  

在大型网站中是离不开消息中间件的，而JMS即Java消息服务是JCP提出的一种消息规范，ActiveMQ全面支持JMS1.1。

部署

从Apache可以下载到最新的Apache ActiveMQ ，然后解压即可，以Windows为例，只需要在安装目录运行如下命令即可。

ActiveMQ\apache-activemq-5.11.1\bin\win64>activemq.bat

默认的监听端口地址是61616并且还提供了一个web管理界面

http://127.0.0.1:8161/admin/用户名和密码是admin/admin
至此基本的部署就完成了。更详细的内容参考官方的Getting Started。

消息机制

首先简单介绍下JMS，所谓JMS就是一个Java消息中间件，目前最新的JMS2.0规范是JCP组织2013在JSR343中提出的，而当前ActiveMQ完全支持JCP在2002年提出的JSM1.1，虽然JSM1.1已经很久远了，但是仍然很重要。JMS的介绍可以参考维基百科Java Message Service，也可以深入阅读对应的JSR标准:JMS1.1(JSR914),JMS2.0(JSR343)

但是简单来说JMS支持两种模式的消息机制：点对点模式（Point-to-point）和发布订阅模式（Publish and subscribe）。
点对点模式是JMS QUEUE模型，而发布订阅模式是JMS TOPIC模型。

点对点模式

发布者将消息发布到对应的Broker（即消息中间件）而存储下来，直到有消费者来获取消息。消息在未被消费之前一直存储而不丢失，但是消息只能被获取一次，例如如果一个发布者发布了消息 M1 M2 M3 M4 而对应的两个消费者来获取消息，则如果消费者1获取了M1和M2那消费者2只能获取M3和M4了，反之亦然。

发布订阅模式

发布订阅模式的发布者发布的消息也会先发布到Broker，而与点对点不同的是，它发送的消息不会持久下来，也即，如果当前没有对应的消费者在监听消息，消息就丢失了。因此，只有消费者事先已经在监听消息的时候才能收到消息，同时另一个与点对点不同的是，所有订阅了（即正在监听对应Topic）消息的消费者都能收到每个消息的一份拷贝。例如如果一个发布者发布了消息 M1 M2 M3 M4 而对应的两个消费者在监听，这两个消费者都会完整的获取到 M1 M2 M3 M4 这四个消息，而之后再过来监听的消费者就不会获取到这些消息，只能获取到之后发送的消息。

示例

ActiveMQ 也同样支持点对应和发布订阅模式，并完全实现了JSM1.1的要求。
要编写ActiveMQ对应的发布者必须要做如下工作：

1.创建消息发送对象，这里必须按照JMS的规范创建对应的ConnectionFactory、Conncection、
Session、Destination以及MessageProducer。

public Publisher() throws JMSException {
    factory = new ActiveMQConnectionFactory("tcp://localhost:61616");
    connection = factory.createConnection();
    connection.start();
    session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
    destination = session.createQueue("FirstQueue");
    producer = session.createProducer(destination);
}

2.然后按照业务逻辑发送消息，这里示例为了说明逻辑比较简单，发送的消息就是一个简单的字符串。

public void sendMessage(String job) throws JMSException {
        TextMessage message = session.createTextMessage(job);
        System.out.println("Dest:"+ destination + "Msg:" + message);
        producer.send(message);
}

3.在使用完资源后进行资源回收和关闭，这也是所有程序在实现的时候都应该做的事情。

public void close() throws JMSException {
    connection.close();
    session.close();
    producer.close();
}

同样的如果编写对应的消费者也和发布者类似的逻辑，代码片段如下：

1.创建消费者对象，这里必须按照JMS的规范创建对应的ConnectionFactory、Conncection、
Session、Destination以及MessageConsumer。

public Subscriber() throws JMSException {
    factory = new ActiveMQConnectionFactory("tcp://localhost:61616");
    connection = factory.createConnection();
    connection.start();
    session = connection.createSession(false, Session.AUTO_ACKNOWLEDGE);
    destination = session.createQueue("FirstQueue");
    consumer = session.createConsumer(destination);
}

2.然后编写接收消息的逻辑，这里是使用同步阻塞的consumer.receive()一直监听消息。

public void recieve() throws JMSException {
    while(true){
        TextMessage  message = (TextMessage)consumer.receive();
        if(message != null){
            System.out.println(message.getText());
        }
    }
}

3.关闭资源

public void close() throws JMSException {
    connection.close();
    session.close();
    consumer.close();
}

如上是创建基于点对点的模式的消息机制：

destination = session.createQueue("FirstQueue");

接收者在创建Destination的时候名字必须是发布者创建的名字，这里是FirstQueue才能正确接收到消息。如果要创建发布订阅模式的消息机制，最简单的方式是：

destination = session.createTopic("FirstTopic");

这样就是创建了一个发布订阅模式的消息机制。

另外：上面的例子中接收者使用的是同步阻塞的处理机制，也就是只有有消息的时候才能处理消息，否则一直阻塞，这样的好处是逻辑简单，但是也因此导致消费者不能在等待消息的时候处理其他任务（比如消息的加工）。其实还可以使用另外的一种非阻塞的方式处理，就是为消费者注册一个MessageListener监听器，负责监听消息，一旦有消息过来就处理消息，而与此同时不影响消费者做其他事情。（显然注册监听器方式和同步recieve()方式不能同时设置）

首先为消费者注册监听器：

consumer.setMessageListener(new MyMsgListener());

然后编写监听器的逻辑，监听器必须实现MessageListener，示例如下：

class MyMsgListener implements MessageListener{
    @Override
    public void onMessage(Message message) {
        TextMessage  msg = (TextMessage)message;
        if(message != null){
            try {
                System.out.println(msg.getText());
            } catch (JMSException e) {
            }
        }
    }

}

完整的代码在：https://github.com/yidi/Learn/tree/master/mq/activeMQ

分别是一个使用监听器的非阻塞方式的发布订阅模式和一个阻塞方式的点对点示例。
